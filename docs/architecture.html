core/

The load-bearing rule: the repository returns raw candidates (ids + fts rank + vec distance), never ranked posts. store knows SQL; search knows ranking; neither imports the other. That single split is what lets a future Postgres migration touch only store/, and lets ranking be tested with no database at all.

2The seam pattern

Every infrastructure dependency follows the same shape. The application depends on the socket (an interface), never on either plug. Production plugs in the real impl; tests plug in a fake.

productionFastEmbedderreal ONNX model, downloads weights, ~5–30 ms/call
Embedder
embed(texts) → vectors
tests / CIFakeEmbedderhash → deterministic vector, no download, instant
▲   the app imports only the socket   ▲

This is why the integration test needs no network and no 30 MB model, and why the static-token → better-auth swap (ADR 0002) never touches a single caller.

6Composition root

One function names the real implementations — buildServer(deps). It is also the integration-test entry point: same function, fakes instead of reals.

// server.ts — the ONLY place implementations are wired
export function buildServer(deps: Deps): { mcp: FastMCP; app: Hono } {
  const query = makeQueryTool(deps)      // tools depend on interfaces,
  const post  = makePostTool(deps)       // never on concretes
  // …confirm, flag
  const mcp = new FastMCP({ authenticate: deps.auth.authenticate })
  mcp.addTool({ name: "query", parameters: zQuery, execute: query })
  // …post, confirm, flag
  const app = mcp.getApp()        // same Hono instance
  mountAuth(app, auth)            // better-auth: sessions + api keys
  mountAdmin(app, deps); mountReview(app, deps)  // JSON API (role-gated + review)
  mountConsole(app)               // serve the built React SPA
  return { mcp, app }
}
production — main.ts 
buildServer({
  repo:  new SqliteRepository(db),
  embedder: new FastEmbedder(),
  auth:  new BetterAuthAuthenticator(auth, repo),
  clock: systemClock,
  idgen: nanoidGen,
}).mcp.start()

Shipped as the Docker image.

integration test

No model, no network, deterministic.

7Data model

Three real tables, two virtual tables, one meta. The event log is the source of truth; last_confirmed is denormalised onto posts purely so ranking avoids a per-query aggregate.

posts 
id            TEXT pk   post_+nanoid
situation     TEXT      embedded + FTS
body          TEXT      FTS
environment   TEXT      embedded
repo          TEXT      ranking boost
status        TEXT      active|retired
created_by    → user
created_at    INTEGER
last_confirmed INTEGER  denormalised
post_events (the log) 
id          TEXT pk
post_id     → posts
verdict     TEXT   confirm|flag
reason      TEXT   incorrect|stale|dup
note        TEXT   optional comment
created_by  → user
created_at  INTEGER

Confirms & flags are events, never bare counters — so distinct-confirmer trust & decay can be added later, recomputed from here.

           synced by SQL triggers

posts_vec  vec0(
  post_id TEXT pk,
  situation_embedding   FLOAT[384],

Hand-written SQL migrations — drizzle-kit doesn’t model fts5/vec0.

9File layout

A pnpm monorepo. Folder colours match the dependency stack (§1): each module is one folder, and interface + real impl + fake live side by side so the seam is obvious on disk.

  • One folder per module. The folder name is the §1 layer; its colour here says which layer it belongs to.
  • Seam = three files together. embedder.ts (interface) · fastembed.ts (real) · fake.ts — same folder, so swapping is local and visible.
  • Tests live next to source. rrf.test.ts beside rrf.ts; the one cross-cutting integration test sits in test/.
  • Raw SQL is quarantined. Only store/queries.ts + migrations/ contain SQL — the seam that a future Postgres move touches.
  • Two “schemas”, never confused. store/schema.ts = Drizzle table defs (drizzle-kit reads it, stays in server — never cross-package, or migrations break). Each tool’s zod input schema lives in its own tools/<name>.ts. Unrelated concerns.
  • Two packages, no shared. server/ ships as a container; agent-plugin/ (markdown + a JSON snippet) ships to humans and imports no TS — so there’s nothing to share. Extract a shared/ package only when a real second TS consumer appears.
